# Release Notes 编写规范

本目录用于存放按版本维护的站内 Release Notes。它们是面向项目已确认
读者的正式版本说明，应帮助研发、测试、运维和交付角色理解版本变化、
升级动作与风险，不得写成无证据的宣传文案。

## 目标读者

编写前先确认宿主项目的实际读者和可见性。内容应优先回答：本版本带来
了什么能力或修复，内部边界发生了什么变化，部署和升级时需要做什么，
以及存在哪些兼容性或回滚风险。

## 编写原则

### 1. 语言与术语一致

沿用宿主项目正式文档的主要语言和术语。代码标识、命令、路径、接口名、
镜像名和版本号应保留可核对的原始值，不做含义改写。

### 2. 保留重要技术事实

不为追求简短而省略会影响开发、测试、部署或交付的变化。有证据时应
分别说明：

- 功能更新与缺陷修复
- 内部架构与代码边界调整
- 数据库结构与迁移链变化
- 部署配置与运维链变化
- 镜像、Chart、安装包等交付资产
- 升级、兼容性、回滚与已知风险

某个类别经证据核对确实无变化时可省略；不得猜测或用“其他改进”替代
未完成的核对。

### 3. 按主题组织，不复制提交流水账

将 issue、PR、commit、测试、部署配置和交付产物等证据按能力主题与部署
主题重新组织。每项结论都应能回溯到真实证据，但正文不应只是提交列表。

### 4. 升级动作必须可执行

需要操作方执行动作时，写明适用前提、命令或文件路径、执行顺序、验证方式与
必要的回滚提示。不得只写“升级到新版本”。

### 5. 发布资产必须可核对

已生成的发布资产应记录真实名称、版本或 tag、位置，以及已完成的验证。
尚未生成或无法确认的资产必须明确标记，不得伪造地址、摘要或通过状态。

### 6. 记录内部收敛与重构

影响运行边界、调试、可观测性、初始化、升级或交付的内部收敛也是版本事实。
只要它会影响目标读者理解、验证或操作该版本，就应纳入对应主题。

## 文件命名

版本页固定使用 `vX.Y.Z.md`，版本号必须来自已确认输入，不加中文标题或
其他后缀。页面 frontmatter 必须满足宿主七字段契约，并使用
`doc_type: release`、`stage: release`；在发布审计完成前保持
`last_verified_version: unverified`。

## 推荐结构

根据真实变更取舍章节，推荐顺序为：

1. 版本标题、发布日期与重点概览
2. 功能更新与缺陷修复
3. 内部架构与代码层调整
4. 数据库与升级说明
5. 部署配置与发布资产
6. 兼容性、已知风险与回滚提示
7. 对比链接与关联范围

## 维护要求

- 编写和核对只使用已确认的版本、变更范围与证据。
- 正文获维护者明确确认前，不得修改 `index.md`、`.meta/releases.json`
  或导航派生面。
- 确认后按版本从新到旧更新 `index.md` 与 release metadata；导航由站点脚本
  生成，不手工维护 sidebar。
- 首次在干净 checkout 验证时，先在 `docs/site` 执行
  `npm ci --ignore-scripts`，再执行 `npm run test:docs`；全部成功后才可输出站内
  Release Notes ready handoff。
- GitHub Release 是后续交付面；本目录内容可作为其已确认输入，但不因此
  授权创建 tag 或 GitHub Release。
